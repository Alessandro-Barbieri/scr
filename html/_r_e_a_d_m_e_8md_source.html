1 ========================================================================
2 Scalable Checkpoint / Restart (SCR) Library
3 ========================================================================
4 
5 The Scalable Checkpoint / Restart (SCR) library enables MPI applications
6 to utilize distributed storage on Linux clusters to attain high file I/O
7 bandwidth for checkpointing and restarting large-scale jobs. With SCR,
8 jobs run more efficiently, recompute less work upon a failure, and reduce
9 load on critical shared resources such as the parallel file system.
10 
11 Detailed usage is provided in the [user manual](/doc/scr_users_manual.pdf).
12

14 Dependencies
15 -----------------------------
16 
17 SCR depends on a number of other software packages.
18 
19 Required:
20 
21  * [pdsh](http://sourceforge.net/projects/pdsh) -- parallel remote shell command (pdsh, dshbak)
22  e.g.:
23  ```bash
24  wget http://downloads.sourceforge.net/project/pdsh/pdsh/pdsh-2.18/pdsh-2.18.tar.bz2
25  bunzip2 pdsh-2.18.tar.bz2
26  tar -xf pdsh-2.18.tar
27  pushd pdsh-2.18
28  ./configure --prefix=/tmp
29  make
30  make install
31  ```
32 
33  * [Date::Manip](http://search.cpan.org/~sbeck/Date-Manip-6.56) -- Perl module for date/time interpretation
34  e.g.:
35  ```bash
36  yum install perl-Date-Manip
37  ```
38 
39  * [SLURM](http://slurm.schedmd.com/) -- resource manager
40 
41 Optional:
42 
43  * [libyogrt](https://github.com/LLNL/libyogrt) -- your one get remaining time library
44  * Enables a running job to determine how much time it has left
45  in its resource allocation.
46 
47  * [io-watchdog](https://github.com/grondo/io-watchdog) -- tool to catch and terminate hanging jobs
48  * Spawns a thread that can kill an MPI process (and hence
49  the job) when it detects that the process has not written
50  data is some specified amount of time. This is very useful
51  to catch and kill hanging jobs, so that SCR can restart the
52  job or fetch the latest checkpoint.
53 
54  * [MySQL](http://www.mysql.com) -- for logging events
55 
56 SCR is designed to be ported to other resource managers, so with some
57 work, one may replace SLURM with something else. Also libyogrt is
58 optional and similarly can be replaced with something else.

60 -----------------------------
61 Configuration
62 -----------------------------
63 
64 Most of the default settings in the SCR library can be adjusted. One
65 may customize default settings via configure options, compile-time
66 defines, or a system configuration file.
67 
68 In particular, the following parameters likely need to be adjusted for
69 other platforms:
70 
71 ### Cache and control directories:
72 
73 It is critical to configure SCR with directory paths it should use
74 for cache and control directories. Each of these directories should
75 correspond to a path leading to a storage device that is local to the
76 compute node. For more information, see the SCR directories section
77 below.
78 
79 ### scr.conf:
80 
81 A number of options can and must be specified for an SCR installation.
82 A convenient method to set these parameters is to create a system
83 configuration file. To help you get started, an example config file
84 is provided in scr.conf. Copy this file, customize it with your
85 desired settings, and configure your SCR build to know where to look
86 for it using the --with-scr-config-file configure option, e.g.,
87 
88  configure --with-scr-config-file=/etc/scr.conf ...
89 
90 ### libyogrt support:
91 
92 If libyogrt is available, specify --with-yogrt[=PATH] during the
93 configure.
94 
95 The function of libyogrt is to enable a job to determine how much
96 time it has left in its current allocation. While libyogrt is not
97 available outside of LLNL, other computer centers often have some
98 mechanism that provides similar functionality. It should be straigt-
99 forward to replace calls to libyogrt with something else. One must
100 simply modify the scr_env_seconds_remaining() implementation in
101 src/scr_env.c to return the number of seconds left in the job
102 allocation.
103 
104 ### MySQL DB Logging:
105 
106 SCR can log events to a MySQL database. However, this feature is
107 still in development. To enable logging to MySQL, configure with
108 --with-mysql. If you would like to give it a try, see the
109 scr.mysql file for further instructions.
110 
111 The compile-time define variables are specified in src/scr_conf.h. One
112 may modify the settings in this file before building to set different
113 defaults. In many cases, one may be able to avoid having to create a
114 system configuration using this method.

116 -----------------------------
117 Build and install
118 -----------------------------
119 
120 Use the standard, configure/make/make install to configure, build, and
121 install the library, e.g.:
122 
123  ./configure \
124  --prefix=/usr/local/tools/scr-1.1 \
125  --with-scr-config-file=/etc/scr.conf
126  make
127  make install
128 
129 To uninstall the installation:
130 
131  make uninstall
132 
133 SCR is layered on top of MPI, and it must be built with the MPI library
134 used by the application. It uses the standard mpicc compiler wrapper.
135 If there are multiple MPI libraries, a separate SCR library must be
136 built for each MPI library.
137 
138 If you are using the scr.conf file, you'll also need to edit this file
139 to match the settings on your system. Please open this file and make
140 any necessary changes -- it is self-documented with comments. After
141 modifying this file, copy it to the location specified in your configure
142 step. The configure option simply informs the SCR install where to
143 look for the file, it does not modify or install the file.
144 
145 ### Example Programs
146 
147 The test programs included in the examples directory will be installed
148 to `${SCR_INSTALL}/share/scr/examples`. They are built in this directory
149 with `make`.

151 -----------------------------
152 SCR directories
153 -----------------------------
154 
155 There are three types of directories where SCR manages files.
156 
157 ### (1) Cache directory
158 
159 SCR instructs the application to write its checkpoint files to a cache
160 directory, and SCR computes and stores redundancy data here. SCR can
161 be configured to use more than one cache directory in a single run.
162 The cache directory must be local to the compute node.
163 
164 The cache directory must be large enough to hold at least one full
165 application checkpoint. However, it is better if the cache is large
166 enough to store at least two checkpoints. When using storage local
167 to the compute node, the cache must be roughly 0.5x-3x the size of
168 the node's memory to be effective.
169 
170 The library is hard-coded to use /tmp as the cache directory. To
171 change this location, do any of the following:
172 
173  * change the SCR_CACHE_BASE value in src/scr_conf.h,
174 
175  * specify the new path during configure using the
176  --with-scr-cache-base=PATH option,
177 
178  * or set the value in a system configuration file.
179 
180 To enable multiple cache directories, one must use a system
181 configuration file.
182 
183 ### (2) Control directory
184 
185 SCR stores files to track its internal state in the control directory.
186 Currently, this directory must be local to the compute node. Files
187 written to the control directory are small and they are read and written
188 to frequently, so RAM disk works well for this. There are a variety of
189 files kept in the control directory. For some of these files, all
190 processes maintain their own file, and for others, only rank 0 accesses
191 the file.
192 
193 The library is hard-coded to use /tmp as the control directory. To
194 change this location, do any of the following:
195 
196  * change the SCR_CNTL_BASE value in src/scr_conf.h,
197 
198  * specify the new path during configure using the
199  --with-scr-cntl-base=PATH option,
200 
201  * or set the value in a system configuration file.
202 
203 ### (3) Prefix directory - on parallel file system
204 
205 This is the directory where SCR will fetch and flush checkpoint sets
206 to the parallel file system.
207 
208 The prefix directory defaults to the current working directory of the
209 application. To override this, the user can set the SCR_PREFIX
210 parameter at run time.
211 
212 -----------------------------
213 RAM disk
214 -----------------------------
215 
216 On systems where compute nodes are diskless, often RAM disk is the only
217 option for a checkpoint cache. When using RAM disk, one should
218 configure the Linux hard limit to be around two-thirds of the amount of memory
219 on a node. This is a high limit, however Linux only allocates space as
220 needed, so it is safe to use a high setting. Jobs which do not use RAM
221 disk will not be negatively affected by using a high setting. This
222 setting must be set by the system administrators.

224 -----------------------------
225 Nodeset format
226 -----------------------------
227 
228 Several of the commands process the nodeset allocated to the job. The
229 node names are expected to be of the form:
230 
231  clustername#
232 
233 where clustername is a string of letters common for all nodes,
234 and # is the node number, like atlas35 and atlas173.
235 
236 More generally:
237 
238 <any non digit characters><any digit characters>
239 
240 so things like atlas-31, atlas31, or even $$##@$!21 are all valid.
241 
242 What is currently not valid is something like:
243 
244 atlas-31vm1
245 
246 because this breaks the regular expressions since it only expects digits
247 characters after the non digit characters. This limitation may be further
248 generalized in the future.
249 
250 The scr_hostlist.pm perl module compresses lists of such nodenames into a
251 form like so:
252 
253  atlas[31-43,45,48-51]
